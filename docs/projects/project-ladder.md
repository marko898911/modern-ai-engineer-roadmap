# Project Ladder

Projects are the spine of the roadmap. Each one should be small enough to
finish, but real enough to show what you learned.

## Ladder

| # | Stage | Project | What it proves |
|---:|---|---|---|
| 0 | Orientation | Learning log | You can track progress and decisions |
| 1 | Foundations | Python data app | You can code, clean data, test, and explain |
| 2 | Machine Learning | ML baseline report | You can train and evaluate honestly |
| 3 | Deep Learning | PyTorch model | You can build and debug a training loop |
| 4 | LLMs | LLM fundamentals notebook | You understand tokens, generation, and model tradeoffs |
| 5 | AI Apps | Evaluated RAG app | You can build and test a useful AI system |
| 6 | Agents | Tool-using agent | You can control an agent loop and trace failures |
| 7 | Infra | Deployed model service | You can operate AI like software |
| 8 | Optimization | Inference benchmark | You can measure and improve latency/cost |
| 9 | Security | AI threat model and ZKML demo | You can reason about trust and verification |
| 10 | Mastery | Capstone | You can own an end-to-end AI system |

## Project Template

Every project should include:

- problem
- user or use case
- dataset or context source
- architecture
- model choice
- evaluation set
- metrics
- implementation
- failure analysis
- security/privacy notes
- next steps

## Portfolio Rule

A portfolio project is not done until another engineer can run it or understand
your results from the README.

Minimum README sections:

```text
# Project Name

## Problem
## Approach
## How to Run
## Evaluation
## Results
## Failures
## Security and Privacy
## Future Work
```

## Beginner-Friendly Project Ideas

| Stage | Idea |
|---|---|
| 1 | Analyze public course reviews and build a CLI recommender |
| 2 | Predict support ticket category and explain errors |
| 3 | Train a small image classifier with augmentation |
| 4 | Compare models on structured extraction tasks |
| 5 | Build a RAG assistant over your class notes |
| 6 | Build a research agent that writes cited summaries and logs tools |
| 7 | Deploy the RAG app with monitoring and eval checks |
| 8 | Benchmark local model serving before and after quantization |
| 9 | Red-team your agent and prove a tiny model decision with ZKML |
| 10 | Combine RAG, agents, deployment, evals, and security into one product |

## Capstone Quality Bar

A strong capstone has:

- a clear target user
- a narrow task
- a realistic architecture
- evaluation before and after improvements
- observability
- security review
- cost and latency numbers
- honest failure cases
