# Roadmap Authoring Standard

Use this standard when adding or expanding pages. It prevents the project from
becoming a complex archive instead of a roadmap.

## Page Rule

Every learning page must answer:

1. Why does this topic matter?
2. What should a beginner learn first?
3. What should they build?
4. What should they measure?
5. How do they know they can move on?

## Required Section Template

Use this structure for new stage-like pages:

```markdown
# Topic Name

## Goal
## Learn
## Build
## Measure
## Exit Criteria
## Common Mistake
## Next
```

## Keep Pages Roadmap-Sized

Pages should guide, not replace books or courses.

Good:

- topic sequence
- important concepts
- project artifacts
- exit criteria
- links to strong resources

Avoid:

- long lecture notes
- giant tool lists
- deeply nested course folders
- vendor-specific details before concepts
- advanced topics with no prerequisites

## Add New Topics Carefully

Before adding a new topic, decide:

| Question | Decision |
|---|---|
| Is it required for every modern AI engineer? | Put it in a stage |
| Is it role-specific? | Put it in tracks or projects |
| Is it fast-moving? | Put it in resources with update notes |
| Is it a tool? | Mention it only after the concept |
| Is it advanced? | Add prerequisites and exit criteria |

## Style

- Use short sections.
- Use tables for paths and comparisons.
- Prefer project outputs over vague advice.
- State what can be skipped.
- Keep file names stable and simple.
- Do not add a new directory level unless it solves navigation.

## Quality Bar

No page is complete until it has a build artifact and exit criteria.
